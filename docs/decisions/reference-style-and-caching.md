# 선택형 스타일 기준(reference) — 추출·전송·캐싱 설계

작성일: 2026-06-08
상태: 분석/추천 (일부 결정, 일부 열림)
관련: [`youtube-coordinate-harvest.md`](./youtube-coordinate-harvest.md)(튜닝용 좌표 확보), [`db-deep-dive.md`](../portfolio/db-deep-dive.md), [`realmysql-experiments.md`](../portfolio/realmysql-experiments.md)(트림 ④)

## 0. 목적 / 배경

운동(예: 스쿼트)마다 **여러 "스타일 기준"**(김종국식, 김계란식 …)을 두고, **사용자가 원하는 스타일을 골라** 그 기준과 비교하며 운동한다. 본 문서는 이 **런타임 기능**의 데이터 모델·추출·전송·캐싱을 다룬다. (오프라인 임계값 튜닝용 좌표 수집은 [`youtube-coordinate-harvest.md`](./youtube-coordinate-harvest.md) 스코프 — 별개.)

> ⚠️ 현재 `ExerciseReference`/`ExtractReferenceData`는 **미구현**(gRPC stub은 빈 응답, `exercise_servicer.py:124`). 본 문서는 그 구현 방향 설계다.

## 1. 결정 / 추천 / 열림 구분

| 항목 | 상태 |
|---|---|
| 운동 1 : N 스타일 기준, 사용자가 선택 | ✅ 결정 (제품 모델) |
| 스타일 1개 = **영상 1개**에서 추출 (N영상 평균 아님) | ✅ 결정 ("1스타일 1영상") |
| 추출은 백그라운드 잡(요청 비블로킹) | 🟢 추천 |
| 기준은 **리샘플 대표 시퀀스**로 저장(원본 프레임 아님) | 🟢 추천 |
| Spring→FastAPI 기준 전송 = unary + async | 🟢 추천 (현 코드와 일치) |
| 기준 좌표 캐싱(카탈로그 패턴) | 🟢 추천 (필수는 아님) |
| 캐시 백엔드(로컬 vs Redis) | 🔶 열림 |
| 라이선스(유튜브 ToS) | 🔶 열림 (리스크) |

## 2. 데이터 모델 — 현재 vs 필요

**현재**(`ExerciseReference`): `exercise_id` + `timestamp_sec` + `joint_coordinates(JSON)`. 즉 **운동당 프레임 행들** = 운동 1 : 기준 시퀀스 1. **스타일 식별자가 없다.**

**필요**(선택형 스타일):
- 기준(스타일) 식별자 도입 — `reference_id`(스타일 단위) 또는 `(exercise_id, style)`. 한 스타일 기준 = 그 시퀀스의 프레임 행들.
- 조회: `findByExerciseId` → **`findByReferenceId`**(사용자가 고른 스타일).
- **세션이 어떤 기준을 골랐는지 기록**(세션–기준 연결) + 사용자 선택(preference)에 현재 스타일 저장.
- 스타일 메타(이름 "김종국식", 썸네일, 난이도 등) 테이블.

## 3. 추출 (영상 → 좌표) — 동시성

스타일 1개 = 영상 1개 → **추출 잡 1개**.

- **백그라운드 잡 (async)**: 추출은 수십 초~분(스트리밍 + MediaPipe). 등록 요청에 **동기로 매달면 안 됨** → 접수 후 즉시 "처리 중" 반환, 워커가 추출·저장, 완료/실패 상태 표시. (트리거 주체가 관리자든 사용자든 동일)
- **병렬은 불필요**: 영상 1개라 코어 분할 의미 없음. 한 영상 내부도 순차 OK. (스타일을 여러 개 한 번에 등록하는 배치라면 그때만 영상 단위 병렬 — embarrassingly parallel.)
- **CPU 바운드 주의**: 추출은 ML이라 CPU 바운드 → 빠르게 하려면 *비동기*가 아니라 *병렬*(multiprocessing). 단일 영상엔 해당 없음.
- 저장: 좌표 **배치 INSERT**, 원본 영상은 **저장하지 않음**(좌표만).

## 4. 저장 단위 — 리샘플 대표 시퀀스 (영상 길이 비의존)

비교(DTW)에 필요한 건 **대표 1렙 패턴**이지 전 프레임이 아니다. `build_reference_sequence`처럼 **rep 분할 → 리샘플(예 ~30프레임) → 대표 시퀀스**로 저장한다.

- **영상이 길어져도 기준 크기는 거의 일정** → 저장·전송 모두 작게 유지.
- `joint_coordinates`는 **트림 13개**([`realmysql ④`](../portfolio/realmysql-experiments.md), 33→13 −60.9%) 적용 — 스타일 N개면 기준 저장량도 N배라 트림 효과가 reference에도 번진다.
- 만약 (잘못) **원본 전 프레임**을 저장/전송하면: 30fps·2KB/프레임 기준 **~2,000프레임(≈66초)에서 gRPC 4MB 한도 초과** → 그때서야 streaming/청크/한도상향 필요. 리샘플하면 이 문제 자체가 안 생긴다.

## 5. 전송 (Spring → FastAPI, 세션 시작 시)

세션 시작 시 사용자가 고른 기준을 `StartAnalysis`로 ai-server에 전달(현 코드 `ExerciseAnalysisService.startAnalysis`).

- **unary + async**: 기준이 리샘플 시퀀스라 **수십 KB** → 4MB 한참 아래 → 한 메시지(`repeated` 필드)면 충분, **streaming 불필요.** 외부 호출이라 `@Async` + async stub(StreamObserver)로 비블로킹(현 코드 그대로).
- **streaming은 한 메시지에 안 들어갈 만큼 커질 때만** — 1스타일 1영상 + 리샘플이면 도달하지 않음.

## 6. 캐싱 — 카탈로그 패턴

기준 좌표는 **세션마다가 아니라 스타일마다 고정**이고, 사용자가 고르는 건 **유한한 스타일 집합**이다 → 상품 카탈로그 캐싱과 동형.

| 속성 | 값 | 캐시 적합 |
|---|---|---|
| 스타일 개수 | 큐레이션된 소수(bounded) | ✅ |
| 각 기준 내용 | 불변(생성 후 고정) | ✅ |
| 재사용 | 인기 스타일을 여러 사용자가 반복 선택 | ✅ |

- **캐시 키 = 기준(스타일) id.** 사용자가 김종국식 선택 → 캐시 조회 → 다른 사용자도 같은 스타일이면 hit.
- **무효화**: 그 스타일 기준이 **(재)생성/수정될 때만** evict — 드물어 간단(write-through 또는 단순 evict).
- **규모**: 스타일 카탈로그가 작으면 **앱 시작 시 전체 preload**도 가능(전부 메모리).
- **백엔드**(🔶 열림): 단일/소수 인스턴스 → 로컬 캐시(Caffeine). 다중 인스턴스 + 일관 무효화 → Redis(+백엔드 포폴 시그널).
- **무의미한 경우와 구분**: 캐싱이 무의미한 건 "매 세션 유니크 일회성 기준(재사용 0)"일 때. **선택형 카탈로그는 그 반대**(유한·불변·공유)라 유효.

### 6.1 구현 옵션 — 캐싱 ≠ Redis

캐싱은 "원본(DB)보다 빠른 곳(메모리)에 복사해 재사용"하는 *개념*이고, **어디에 두느냐**로 구현이 갈린다. Redis는 그중 하나(분산 캐시)일 뿐.

| 구현 | 무엇 | 특징 |
|---|---|---|
| `Map`/`ConcurrentHashMap` 직접 | 프로세스 내 수동 캐시 | 만료·크기 직접 관리 → 비추 |
| **Caffeine** (로컬) | TTL·최대크기·LRU 자동 로컬 캐시 | **가장 빠름(네트워크 0)·단순**, 단 인스턴스별로 따로 |
| **Spring Cache** (`@Cacheable`) | 메서드 결과 캐시 추상화 | 백엔드(Caffeine/Redis)를 설정만으로 교체 |
| **Redis** (분산) | 외부 캐시 서버, 모든 인스턴스 공유 | 일관·공유, 단 네트워크 홉 + 서버 운영 비용 |

**로컬 vs Redis 선택 트리거 = "앱 인스턴스 수"** (유저 상호작용 양이 아님):

| 서버 | 권장 | 이유 |
|---|---|---|
| **1대 (단일 인스턴스)** | **로컬 Caffeine** | 유저 많아도 read 부하는 로컬 캐시가 흡수. Redis 불필요 |
| **여러 대 (수평 확장)** | **Redis** | 로컬은 인스턴스별로 캐시가 갈라져 중복·불일치·무효화 전파 문제 → 공유 캐시 필요 |

> 흐름: 유저/트래픽 증가 → 서버 수평 확장(여러 대) → 로컬 캐시가 인스턴스별로 갈라짐 → **그때 Redis**. 유저 증가는 간접 원인, **직접 트리거는 "다중 인스턴스"**. (Redis는 캐시 외에도 세션 공유·분산 락·rate limit·pub/sub 등 "인스턴스 간 공유 상태"에 두루 쓰임 — 모두 다중 인스턴스라 필요해지는 것.)

**코드 패턴 (Spring Cache — 백엔드 교체 가능):**
```java
@Cacheable("referenceCoords")                      // 첫 호출 DB, 이후 캐시
public List<ExerciseReference> findByReferenceId(Long refId) { ... }

@CacheEvict(value="referenceCoords", key="#refId") // 기준 (재)생성·수정 시 무효화
public void updateReference(Long refId, ...) { ... }
```
→ **로컬 Caffeine으로 시작**했다가, 수평 확장 시 **설정만 Redis로 교체**(코드 불변)하는 승격 경로가 깔끔.

## 7. 정직 캐비엇

- **캐싱은 필수 아님**: 기준 좌표는 작고 인덱스 조회라 DB 읽기 자체가 이미 빠르다. 캐싱의 가치는 "DAU 1,000 가정 하 동시 세션 시작의 **중복 동일 읽기 제거** + 시그널"이지, *측정된 핫 병목 해결*이 아니다. 오버셀 금지.
- **DAU 가정**: 동시성·캐싱 정당화는 "DAU 1,000 가정"([`FINAL-REPORT §5`](../FINAL-REPORT.md)) 기준. "작아서 드묾"으로 회피하지 않는다.
- **라이선스**(🔶): 유튜브 ToS는 비공식 경로 추출(다운/스트리밍 무관) 사실상 금지. 좌표만 저장해도 ToS 위반은 남고 저작권도 회색. → **자체 촬영/CC 영상 우선**, 유튜브는 보조, 발표물에 "유튜브 수집" 박지 말 것.
- **선행**: `ExtractReferenceData` 구현 필요(현 stub). ai-server는 별도 설계라 [Python 변경 전 확인](../../) 원칙 적용.

## 8. 한 줄 요약

운동 1 : N 스타일 기준(사용자 선택, 스타일당 영상 1개). 추출=백그라운드 잡(영상 1개라 병렬 불요), 저장=리샘플 대표 시퀀스+트림(영상 길이·크기 비의존), 전송=unary+async(현 코드), 캐싱=카탈로그 패턴(키=스타일 id, 필수 아닌 최적화+시그널). 데이터 모델에 스타일 식별자 도입이 선행.
