## 0. 한 줄

> 원래 "1초 평균 집계 = 저장공간 감축"으로 적었으나, 서사가 **raw=단기 버퍼(TTL)·derived=영속**(§2-0)으로 진화하면서 *영속 공간* 명분은 이미 해소됐다. **남은 진짜 효용은 "라이브 적재(write-heavy) 비용 감소" 하나**이며, 그게 사실이라면 **어디서(AI vs Spring) 다운샘플링하느냐**가 결정 대상이다.

## 1. 현재 흐름 (실코드 — 프레임당 1행)

```
폰 ──(프레임마다 base64 POST)──▶ FastAPI /pose
   MediaPipe detect → analyzer.process_frame
   visibility 통과한 매 프레임 → state.current_rep_frames.append(...)   # pose.py:91-96  ← 1프레임=1행
   rep 완성 시 → 그 rep의 모든 프레임을 batch gRPC 전송               # pose.py:107-116
        │  report_pose_data_batch(session_id, [PoseDataRequest ×N])
        ▼
   Spring PoseDataService.savePoseDataBatch
        jdbcTemplate.batchUpdate(INSERT ... )  # 1프레임=1 INSERT row     # PoseDataService.java:54
        ▼
   MySQL pose_data  (rep당 ~5~30행, §roadmap-5)
```

**확정 사실**: `pose_data`는 **프레임 단위로 1행씩** 쌓인다. "1초 평균/다운샘플"은 **현재 미구현**(roadmap §7 갭). DAU 1,000·1억 행 합성은 이 프레임 단위 가정 위 숫자다.

> 중요 분리: **라이브 분석(DTW·sync·rep 감지)은 저장 전 FastAPI에서 끝난다.** `pose_data`에 들어가는 건 *리포트(precompute)용 사후 데이터*. → **저장본을 다운샘플해도 라이브 분석 정확도엔 영향 없고, 영향받는 건 리포트 시계열 해상도뿐.** (핵심 안전판)

## 2. 재구성 — 다운샘플링이 *지금* 무엇을 위한 건가

원래 "1초 평균"이 뭉뚱그린 3가지 관심사를 분리하면:

| 관심사 | §2-0 진화 후 상태 | 다운샘플링이 필요? |
|---|---|---|
| **A. 영속 저장공간** (연 ~800GB raw 오염) | ✅ **이미 해소** — raw는 단기 TTL 버퍼(DROP PARTITION), derived만 영속(precompute) | ❌ 불필요 (TTL이 해결) |
| **B. 라이브 적재 비용** (write-heavy: INSERT 행수·gRPC payload·풀 압박) | 🔶 미해소 — 프레임당 1행이 그대로 DB·네트워크 부하 | ✅ **여기서만 유효** |
| **C. 리포트 시계열 해상도** | 현행 프레임 단위(과할 수 있음) | ⚠️ 다운샘플 시 해상도 trade |

→ **결론: 다운샘플링의 유일하게 살아있는 명분은 B(라이브 적재 비용).** 저장공간(A)으로 정당화하면 §2-0과 모순되니 **B로만 정당화할 것.** 그리고 B가 실제로 충분히 큰 문제인지부터가 사전 질문(§5 측정).

## 3. 결정 대상 — 어디서 다운샘플링하나

B(적재 비용)를 줄이기로 *가정*하면, 줄이는 지점은 셋:

| | A. FastAPI(AI)에서 | B. Spring INSERT 직전 | C. 안 함 (raw 그대로 + TTL) |
|---|---|---|---|
| 위치 | gRPC 전송 *전* 프레임 평균/추출 | `savePoseDataBatch` 안에서 평균 후 batchUpdate | 다운샘플 없음 |
| **gRPC/네트워크 절감** | ✅ 전송량 자체 감소 | ❌ 이미 다 받은 뒤 | ❌ |
| **DB INSERT 행수 절감** | ✅ | ✅ (실측 병목은 DB INSERT — §realmysql ②a) | ❌ |
| **ai-server 수정** | 🔴 **필요** (타인 설계, 확인 필수 [[feedback_minimize_python_changes]]) | 🟢 불필요 | 🟢 불필요 |
| **쓰기 축 서사 소유** | AI 쪽으로 분산 | 🟢 **Spring이 적재 최적화 소유**([[user_career_target]] 백엔드 포폴) | — |
| 리포트 해상도 통제 | AI가 정책 보유 | 🟢 Spring(저장 계층)이 정책 보유 | 원본 유지 |
| 라이브 분석 영향 | 없음(분석은 저장 전 완료) | 없음 | 없음 |

### 3-A. FastAPI에서 (네트워크까지 절감, 단 ai-server 침습)
- 장점: gRPC payload·DB 행수 **둘 다** 감소 — 가장 근본. 폰→AI는 어차피 못 줄이지만 AI→Spring 구간을 줄임.
- 단점: ai-server는 **타인 설계** → 무단 수정 금지, 협의 필요. squat 분석 로직과 같은 파일(`pose.py`)이라 결합 위험.
- 포지셔닝: 절감 효과는 최대지만 **백엔드(Spring) 포폴 서사로는 약함**(최적화가 AI 쪽 코드에 있음).

### 3-B. Spring INSERT 직전에서 (추천 후보 — 백엔드 소유)
- 장점: **ai-server 무수정**, 적재 최적화를 **Spring(쓰기 축)이 소유** → 포폴 서사 정합. 실측 병목이 **DB INSERT**(§realmysql ②a: batch로 throughput +99%)였으므로, 행수 감축은 *그 병목*을 직접 친다.
- 단점: gRPC/네트워크는 **이미 다 받은 뒤**라 절감 못 함(AI→Spring 구간 낭비 잔존). Spring CPU로 평균 계산.
- 확장성([[feedback_industry_level_standard]]): 동시 세션↑ 시 INSERT 행수가 풀·p99을 압박 — 여기서 감축이 효과. 단 네트워크가 병목이 되는 구간에선 B로 부족 → 그때 A 재고.

### 3-C. 안 함 (정직한 baseline)
- §2-0가 영속 공간을 이미 해결하므로, **적재 비용이 측정상 문제 아니면 다운샘플은 premature**([[redis-introduction]] 기조: 측정 전 최적화 금지).
- raw를 짧은 TTL 버퍼로 두고 DROP PARTITION으로 버리면, "프레임당 1행"이어도 영속 부담 없음. **B(적재 비용)가 실측으로 아플 때만** A/B 착수.

## 4. "1초 평균" vs "1초 1프레임 추출" (방식 자체)

다운샘플을 한다면 *어떻게*도 갈린다:
- **평균(average)**: 1초 구간 N프레임의 좌표를 평균 → 1행. 노이즈 완화하나 **JSON 좌표 평균의 의미가 모호**(관절 위치 평균이 실제 자세 아님), sync_rate·feedback 같은 비수치 필드 병합 애매.
- **대표 추출(decimate/pick)**: 1초당 1프레임만 남김(예: 구간 중앙/최악 sync). 의미 보존·단순. **리포트가 "구간 대표"면 충분**이라 보통 이쪽이 정직.
- 추천 경향: "평균"이라는 이름에 갇히지 말고, **리포트가 요구하는 해상도에 맞춘 대표 추출**이 보통 더 안전(특히 worst-rep 분석은 평균이 아니라 극값이 중요 — §realmysql 리포트 집계).

## 5. 측정 먼저 (착수 전 — 말 아닌 증거)

B가 실제 문제인지부터 ghz rig로 확인(이미 보유):
- 현행 프레임당 1행에서 **동시 세션 N(→DAU 1,000 환산)별 INSERT throughput·p99·HikariCP active/pending** — §realmysql 배치 카드 재사용.
- 다운샘플(예: 1초 1행, ~Mx 감축) 적용 후 동일 측정 → **행수 1/M ↔ p99·풀 개선폭**.
- 네트워크: AI→Spring gRPC payload 크기 before/after (A안일 때만 의미).
- 판정: 감축이 p99/풀을 의미 있게 낫게 하면 착수, 아니면 C(안 함) 유지가 정직.

## 5-1. 측정 결과 (2026-06-12 실측) ⭐

rig: `loadtest/.bin/ghz.exe` → `SavePoseDataBatch`, 백엔드 :6565(JDBC batch 빌드), `pose_data` 실데이터 412만 행 위, 워밍업 후 각 n=500~600. **단일 머신 동거(4코어) — 절대 RPS는 오염, 신뢰는 *형태/상대*.**

### (1) baseline — 동시성 천장 ~25 RPS, p99 폭증
단일세션(801) 고정, c 스윕: RPS는 c=5~10에서 **~25에 평탄**, p99는 c=5 566ms → c=70 **11.6초**. 처리량 평탄 + 지연 폭증 = 포화. err=0(안 터지고 느려짐). ≈625 행/초 천장.

### (2) 병목 귀속 — 풀=10이 진짜 캡, 버퍼풀 가설은 반증
c=50 지속 부하 중 MySQL 내부 관측:
- **디스크 읽기 +2회**(45초간), 논리읽기 +34만 → **전부 인메모리. I/O 바운드 아님.** "버퍼풀 128MB라 디스크 친다" 가설 **반증**(단일세션이라 작업셋이 작아 풀 상주).
- **backend 커넥션 10개 전부 `Query`** = HikariCP 풀=10(Spring 기본). c↑여도 DB엔 항상 10개만 → **풀이 throughput 캡**.
- `data_lock_waits`=0(append라 행-락 무). `threads_running`≈11인데 CPU 1.5/4코어만 → **내부 직렬화 대기**(단일 session_id=801 인덱스 리프 래치·redo 커밋).
- **결론**: 천장 = ①풀=10 + ②단일세션 핫스팟(rig 결함) + ③동거. **INSERT 행수/디스크 아님.**

### (3) 다세션 rig — 천장은 그대로, 핫스팟은 꼬리지연만 부풀렸음
901~1900 세션 round-robin(아티팩트 #2 제거):

| c | 단일 RPS / p99 | 다세션 RPS / p99 |
|---|---|---|
| 30 | 19.7 / 6341 | 27.1 / **2159** |
| 100 | 22.7 / 10731 | 26.4 / **5004** |

→ **RPS 천장 불변(~25-29) = 풀이 캡**. 단일세션은 **p99만** 2~3배 부풀렸음(실제 다세션 워크로드는 꼬리지연 덜 나쁨).

### (4) R-sweep — 배치 비용은 고정비용 지배 → 다운샘플은 1순위 처방 아님 ⭐
배치당 행수 R을 25→5→1로(다세션, 풀=10):

| c | R=25 | R=5 | R=1 |
|---|---|---|---|
| 10 | RPS 29.7 / **742행/s** / p50 300 | RPS 48.3 / 242 / 195 | RPS 58.7 / 59 / 165 |
| 50 | RPS 17.8 / **445행/s** / p99 8007 | RPS 40.3 / 202 / 2363 | RPS 69.8 / 70 / 1068 |

- **고정비용 지배(sub-linear)**: R 5배 줄여도 배치지연 W는 5배가 아니라 ~1.8배만↓(1970→1094ms). rows/s는 폭락(742→59 at R1) → 배치 지연은 행수가 아니라 라운드트립·커밋·풀획득 같은 **고정비용이 대부분**. (batch insert +99%의 역증명 — 배치가 고정비용 분산.)
- **그런데 그 sub-linear 개선조차 처리량을 크게 올림**: 처리량=풀/W 라 W 1.8배↓ → **RPS c=50서 +126%(17.8→40.3), p99 −70%(8007→2363)**. 즉 "행수엔 둔감하지만 둔한 개선이 처리량을 배로." → **다운샘플은 진짜 쓰기축 레버**(앞서 "~35%·부수카드"는 저부하 p50만 본 오판, 정정).
- **수확체감 = 스위트스폿 ~R5**: R 5→1은 행 또 5배 줄여도 처리량 40→70만(rows/s 폭락) → **R=1 같은 과도 감축은 행당 최악.** "1초 평균"식 극단 금지 근거.

### (5) 풀 사이징 실측 — 천장은 풀이 아니라 **박스(2 물리코어)** ⭐ 반전
풀=30 컨테이너(별 포트) 재생성 후 다세션 동일 스윕:

| c | 풀=10 RPS / p99 | 풀=30 RPS / p99 |
|---|---|---|
| 10 | 29.4 / 1338 | 22.4 / 2857 |
| 50 | 20.6 / 4873 | 23.3 / 4504 |
| 100 | 26.4 / 5004 | 30.2 / 5212 |

- **풀 3배(10→30)에도 RPS 개선 0**(오히려 저~중 c에서 약간 악화), p99도 악화 → **천장은 풀이 아니라 박스 한계.** 풀=10 "10개 다 바빴음"은 *원인 아닌 증상*(박스가 그 이상 못 밀어냄).
- 추가 20커넥션 = 2코어에 컨텍스트 스위칭만↑(예측한 트레이드오프). HikariCP 스위트스폿 (물리2×2)+1≈5 → pool=10 이미 위, 30은 과함.
- **머신 한계 확인** = 동시성(30)을 늘려도 병렬성(2코어) 불변 → 처리량 못 늘고 꼬리지연만. 진짜 레버는 풀숫자가 아니라 **코어↑ / 수평확장**. ⚠️ 단 이 절대천장 자체가 **i3 2코어 동거 측정**이라 무의미([[project_loadtest_env_constraint]]) — 신뢰는 "풀↑이 안 통한다"는 *상대 결론*뿐.
- ⚠️ **추가 정정(2026-07-11)**: HikariCP 스위트스폿 공식(코어×2+스핀들)이 원래 가리키는 대상은 **DB 서버의 코어수**다 — 커넥션을 그 이상 열어봐야 DB가 동시에 처리할 수 있는 병렬도 이상으로는 대기열만 늘어난다는 논리. 이 실험은 mysql·backend가 **같은 물리 2코어를 공유**하는 로컬 환경이라, "코어"가 DB 서버 코어인지 백엔드(JDBC 클라이언트) 코어인지 구분이 안 된 채 "박스가 병목"으로 뭉뚱그려졌다. 실제 프로덕션처럼 앱 서버와 DB가 **분리된 인스턴스**(RDS 등 관리형 DB가 표준)라면 백엔드 코어수는 커넥션 풀 사이징과 무관해지고, DB 서버 자체의 코어·디스크 기준으로 재계산·재실측해야 한다 — 지금 "pool=10이 충분하다"는 결론은 **이 동거 환경에 종속**된 것이지 분리된 배포에 그대로 이전 가능한 결론이 아니다.

> 스크립트: `loadtest/ghz/ceiling.ps1`(단일), `gen_batch_multi.py`(다세션 array), 결과 `results/multi-c*.json`·`rsweep-c*-r*.json`·`pool30-c*.json`. 풀30 컨테이너는 일회용(측정 후 제거).

### (6) 분리 배포 가정 시 풀 사이징 — 이론 계산 (2026-07-11, 실측 아님)

**목적**: (5)의 정정에서 지적했듯, "pool=10 충분"은 mysql·backend가 같은 2코어를 공유하는 로컬 환경 종속 결론이다. 실제 프로덕션처럼 앱 서버와 DB가 분리된다면 재계산이 필요한데, 지금 이 프로젝트는 AWS에 실제 배포할 계획이 없어([[project_keep_server_ai_architecture]] 계열 방침과 동일한 이유로 실비용 인프라 전환은 안 함) 실측 대신 **가정 기반 이론 계산**으로 정리한다. §2-3-1/2-3-2(GPU 도입 사고실험)와 같은 패턴 — 실제 착수 대상이 아니라 면접 설계 서사용.

**가정**: 타깃 RDS 인스턴스 = `db.t4g.micro`(2 vCPU, Graviton, 버스터블). 근거: 이 프로젝트가 이미 못박아둔 DAU 1,000 가정([[feedback_state_assumption_design_to_it]])에 맞는 최소 현실적 등급 — 그 이상을 가정하면 과설계.

**계산**: HikariCP 스위트스폿 공식 = (코어수 × 2) + 유효 스핀들수. `db.t4g.micro` = 2 vCPU, RDS 스토리지는 SSD(gp3) 기반이라 스핀들=1로 단순화하면:

```
(2 × 2) + 1 = 5
```

**흥미로운 우연**: 로컬 실험(5)에서 나온 "pool=10 이미 스위트스폿(≈5) 위"의 그 5와 **숫자가 같다**. 하지만 이유는 완전히 다르다:
- 로컬: 2코어를 **backend와 mysql이 나눠 쓰는** 환경이라 그 2코어가 병목이었음(백엔드 JDBC 클라이언트 작업과 MySQL 서버 작업이 같은 CPU를 두고 경쟁).
- `db.t4g.micro` 가정: 2 vCPU는 **DB 전용**(앱 서버는 별도 인스턴스) — 백엔드가 뭘 하든 이 코어 수와 무관, 순수하게 "RDS가 동시에 처리할 수 있는 쿼리 병렬도"만 반영한 숫자.

**시사점**: `db.t4g.micro`에 배포한다고 가정하면 현재 코드에 명시된 `maximum-pool-size=10`은 이론상 스위트스폿(5)의 2배 — 로컬 실험이 보여준 "더 큰 풀도 해 안 됨, 컨텍스트 스위칭만 증가" 패턴이 여기서도 반복될 가능성이 있다는 뜻(단, 이번엔 병목 주체가 DB 서버 코어라는 점이 다름).

**정직하게 짚을 점 (실측 아님을 다시 강조)**:
- 이 계산은 **`db.t4g.micro`를 실제로 띄워 부하를 걸어본 게 아니다** — 순수 공식 대입.
- `db.t4g.micro`는 버스터블(T-series) 인스턴스라 CPU 크레딧이 있고, 베이스라인(약 10~20%) 초과 지속 시 스로틀링된다 — 이건 풀 사이징과는 별개의 제약이라 공식만으로는 안 잡힘.
- 공식 자체가 "짧은 트랜잭션형 쿼리" 워크로드를 전제로 함 — 이 프로젝트는 대체로 단순 조회·배치 INSERT라 전제에 부합하지만, 장시간 트랜잭션이 섞이면 공식이 안 맞을 수 있음.
- 이 이론 계산이 실제와 어긋나는지 확인하려면 결국 실제 `db.t4g.micro` RDS를 띄워 실측해야 하는데, 이건 실비용이 드는 별도 결정이라 착수하지 않음(사용자 confirm 대기 상태로 남김).

## 6. 추천 (2026-06-12 측정 완료 — 결정 전)

측정이 가설을 **두 번 수정**시킴: ① "행수가 아픔" → 아니다(풀/박스) ② "풀=10이 캡" → 아니다(박스). 최종 귀속: **천장은 i3 2 물리코어 박스 한계**(풀↑도 안 통함). 단 그 절대천장은 동거측정이라 무의미 — **신뢰는 메커니즘/상대뿐**([[project_loadtest_env_constraint]]).

| 레버 | 측정 결과 | 결론 |
|---|---|---|
| 풀 사이징(10→30) | RPS 개선 0, p99 악화 | ❌ **천장 해법 아님**(박스-바운드). 이미 스위트스폿 위 |
| 다운샘플(R 25→5) | RPS +126%·p99 −70%(c=50), 저장행 5배↓ | 🟢 **진짜 레버** — 배치당 W를 줄여 처리량·지연·저장 동시 개선. 단 R≈5 스위트스폿(R=1 과도) |
| 수평확장 / 코어↑ | (미측정) | 박스가 진짜 천장이므로 절대용량의 답. 단 DAU 규모상 당장 불요 |

> 핵심 한 줄(최종): **"측정으로 가설을 두 번 깼다 — 쓰기 천장은 행수도 풀도 아닌 2코어 박스였다(풀 30도 무효). 다운샘플은 배치당 비용을 줄여 처리량 +126%·저장 5배↓를 동시에 주는 진짜 레버지만, 절대 천장은 이 로컬로 못 재고 박스/수평확장이 답이다."** — *가설→측정→반증을 두 번 돌려 레버를 재배치*한 게 서사 핵심.

**다운샘플 착수 판단**: 성능(쓰기 천장)이 아니라 **저장 감축 + 배치 효율 + 헤드룸** 목적이면 가치 있음. 위치 B(Spring 대표추출), R 하한 ~5, 리포트 해상도 SLA 확인 후. "성능 처방"으로 과장 금지.
